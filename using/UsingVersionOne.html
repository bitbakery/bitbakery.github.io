Using VersionOne

VersionOne is a web-based project management and collaboration tool. There are often many ways to accomplish the same thing within VersionOne. In this cheat sheet, we will focus on a minimal subset of the available views, but as you become more comfortable with the tool, you may wish to explore some of the other views and customization options to create the environment which works best for you.

Assigning a Backlog Item to a Release Candidate

At PTC, we have chosen to model product releases and their associated release candidates (as well as ongoing maintenance streams) by way of the project hierarchy: Thus, to move an epic (or a story) to a particular release candidate, simply set its project (e.g., to assign this epic to X-20 Release Candidate 1, set its project as "X-20 RC1"): Note, however, that moving an epic to a different project does NOT automatically move any child stories of that epic!

Assigning a Story to a Sprint

VersionOne enables a variety of ways to assign a story to a sprint. The most direct way is to simply set the sprint within the story itself: Adding Tasks to a Story VersionOne provides a variety of ways to add a task to a story. One way to easily add multiple tasks to multiple stories within a sprint (as you might like to do during a sprint planning session) is to do so within the "Sprint Tracking" view: Clicking "Add Task" displays the Task dialog box: Split a Story Occasionally during sprint planning we would like to take some tasks that were defined as part of an existing story and break them out into a new, separate story. This can be done by "splitting" the story: Clicking "Split" brings up the Split dialog box: Splitting a story at the beginning of a sprint can sometimes be a helpful way of managing the sprint backlog, but - in general - if a team is nearing the end of a sprint and it looks as though a particular story won't get completed, it's best to simply move the story to the next sprint, as opposed to splitting the story. Splitting stories at the end of a sprint is usually done for the wrong reasons (i.e., trying to boost a team's numbers, instead of proactively managing a team's sprint backlog).

View Release Candidate Burnup

After clicking the "Go" button, a burnup chart will be generated showing the progress made against the epics assigned to the currently selected project node: Note that when you're generating release or release candidate burnup charts, it's very important that you have the correct project node selected in order to properly specify the collection of stories against which you wish to track progress. To illustrate the point, if you want to generate a burnup chart for the first release candidate of X-20 but you have the "X-20" project node selected instead of "X-20 RC1", you'll see a report of progress made against the entire release backlog, instead of the backlog for that particular release candidate.
